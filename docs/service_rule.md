
# WMS Dispatch 核心业务逻辑说明文档（不可变）

本说明文档定义了 WMS 调度系统中最核心的状态机流转、库存防超卖机制、阶段串行校验以及异常处理准则。所有前后端 API 及 AI Agent 均**必须**严格遵守此文档定义，以确保数据一致性与系统安全。

## 0. 全局时间基线（新增）

系统所有业务时间统一按中国北京时间（`Asia/Shanghai`，UTC+8）执行，包含：

* 数据库默认时间生成
* 业务状态流转时间写入（接单、完成、取消、终止等）
* 超时判断（24h、48h、deadline）
* 前端时间展示与排序

认证例外：JWT 过期时间使用 UTC 时间戳，不属于业务时间口径。

## 1. 订单状态机 (Order State Machine)

订单（`orders`）的生命周期包含四个状态，流转路径极其严格：

* **待接单 (`pending_acceptance`)**
  * 管理员创建订单后的初始状态。
* **进行中 (`in_progress`)**
  * **触发**：调度员认领接单。
  * **系统动作**：校验库存充足后，状态变更为 `in_progress`，记录 `accepted_at`，并将订单所需数量加入商品库存的“预留量（`qty_reserved`）”。
* **已完成 (`completed`)**
  * **触发**：当且仅当该订单的 **“发货 (shipping)”阶段被标记为完成**时（无论是自动还是手动触发）。
  * **系统动作**：正式扣减物理现存量（`qty_on_hand`），同时释放预留量（`qty_reserved`），记录 `completed_at`。
* **已取消 (`cancelled`)**
  * **待接单期的取消**：仅管理员可取消。
  * **进行中期的取消（严格限制）**：仅责任调度员可取消。取消前必须先将该订单下所有 `pending` / `in_progress` 工单逐一手动终止 (`terminated`)。
  * **系统动作**：记录 `cancelled_at`、`cancelled_by` 和取消原因，同时**释放预留库存**，并向管理员推送取消通知（当前单管理员场景写入 1 条通知记录）。
  * **幂等要求**：同一订单重复执行取消操作时，系统必须拒绝或返回无副作用结果，禁止重复释放库存。
  * **原子性要求**：订单状态更新、库存释放、通知写入、审计记录必须处于同一数据库事务中，任一环节失败需整体回滚。

**⏳ 超时熔断机制（接单后未派工）**：
接单后 **24 小时内**，若系统检测到该订单未创建任何工单记录（`work_orders`），则触发自动回退逻辑：订单状态重置为 `pending_acceptance`，释放预留库存。

回退审计要求（`orders`）：
* `timeout_revert_count`：每发生一次熔断回退即 `+1`。
* `last_reverted_at`：记录最近一次熔断回退时间。

告警路由分支：
* **场景 A：无人接单超时**（订单长期停留在 `pending_acceptance` 且 `dispatcher_id IS NULL`）：仅通知管理员。
* **场景 B：已接单未派工超时**（订单在 `in_progress` 且接单 24 小时未生成工单）：通知责任调度员和管理员。

## 2. 阶段串行与控制规则 (Stage Sequential Rules)

订单的履约过程由 `order_stages` 表管理，遵循**宏观严格串行、微观允许并行**的原则：

* **阶段顺序**：拣货 (`picking`) → 备货 (`staging`) → 发货 (`shipping`)。
* **初始化时机（固定口径）**：订单被接单进入 `in_progress` 时，一次性初始化三条阶段记录（`picking` / `staging` / `shipping`），初始状态均为 `not_started`。

* **阶段完成的两条路径（核心校验）**：
  1. **自动完成**
   - 某阶段下所有关联 `work_orders` 的 status 均为 `completed` 时，系统将自动检测下一阶段(如果有下一阶段)是否有至少一张工单，如果没有则不自动完成；反之，如果有至少一张工单或没有下一阶段，则将该阶段的 status 置为 `completed`，`completion_type` 记为 `auto`，`completed_at` 记录当前时间，`completed_by` 为空。
   - 创建工单后也检测一次上一阶段（如果有上一阶段）的工单是否全部完成，如果是则自动完成上一阶段以实现逻辑闭环。
  2. **手动标记完成 (`manual`)**：调度员认为已有工单成果足以推进流程时，可主动关闭该阶段。后端 API 必须进行双重强制校验：
     * **条件 1**：该阶段至少有 **1 张**工单状态为 `completed`。
     * **条件 2**：该阶段**没有任何**进行中 (`in_progress`) 或待处理 (`pending`) 的工单。（若有，系统强制拦截，调度员必须先将剩余未完工单手动设为 `terminated`）。
     * **记录**：强制完成必须记录操作人与 `remark`（原因）。
* **⛔ 关门法则 (Lockdown)**：任何阶段一旦其 `status` 变为 `completed`，**系统在物理层面禁止对该阶段创建任何新工单**。

## 3. 工单状态机与执行逻辑 (Work Order Rules)

单张工单（`work_orders`）负责微观执行，同一阶段内可存在多张工单（分配给不同工人）。状态包含四个：

* **阶段归属**：每张工单通过 `stage_id` 强关联到 `order_stages.id`，阶段类型由关联阶段记录提供，不在工单表冗余存储。

* **待处理 (`pending`)**：调度员或 AI Agent 成功派单，记录 `created_at`（即派发时间）。
* **进行中 (`in_progress`)**：工人通过终端点击“开始执行”，记录 `started_at`。（通过计算 `started_at` - `created_at` 评估响应时效）。
* **已完成 (`completed`)**：工人提交完工动作与备注。
* **已终止 (`terminated`)**：调度员主动中止该任务。**注意：工单没有“取消”概念，只能被终止。**

## 4. 库存锁定与高并发扣减 (Inventory Logic)

库存表（`inventory`）是系统数据一致性的最后防线，绝不允许出现“超卖”或“幽灵库存”。

* **口径定义**：
  * `qty_on_hand`：物理现存量（实物未离库，此数值绝对不减）。
  * `qty_reserved`：接单预留量（订单占用的虚拟份额）。
  * `qty_locked`：调拨软锁定量（调拨单占用的虚拟份额）。
  * **`qty_available`（动态可用量）**：由 PostgreSQL 底层通过生成列（`GENERATED ALWAYS AS (qty_on_hand - qty_reserved - qty_locked) STORED`）自动计算，**后端代码严禁对该字段进行写入操作**。

* **业务动作对库存的改变矩阵**：
  * **接单时**：只做 `qty_reserved += N`。
  * **接单超时 / 订单取消**：释放预留 `qty_reserved -= N`。
  * **订单发货完成**：实物离库，双扣 `qty_on_hand -= N` 且 `qty_reserved -= N`。

* **并发控制口径（库存关键路径）**：
  * 一致性保底由数据库约束负责（如非负校验、可用量约束），确保不会超卖。
  * 悲观锁（`SELECT ... FOR UPDATE`）作为推荐策略，用于减少并发下无效 `UPDATE + ROLLBACK`，并让库存不足更早返回。
  * 即使不使用悲观锁，仍不得突破数据库约束；但在高并发接单/取消场景下会带来更高冲突成本与更差用户体验。

## 5. 跨仓调拨流转 (Cross-warehouse Transfer)

跨仓调拨（`transfer_orders`）采用严格的“两段式锁定与二次校验”模型：

1. **申请与软锁定**：调度员提交申请，在源仓库立即创建软锁定（`qty_locked += N`）。此部分数量对其他尝试接单的调度员立即可见为“不可用”。
2. **审批与二次校验**：管理员审批期间，物理库存可能因人工盘点发生变化。管理员点击“通过”瞬间，系统执行二次校验：`当前源仓库 qty_on_hand - qty_reserved - qty_locked >= 0`。若不足，拒绝通过，要求驳回或部分调拨。
3. **出库与在途生成**：审批通过后，源仓库物理现存量真实扣减且软锁定释放（`qty_on_hand -= N`, `qty_locked -= N`）。同时，目标仓库自动生成待入库记录（`inbound_records`）。
4. **入库确认**：目标仓调度员清点实物后确认入库，目标仓 `qty_on_hand += N`。
5. **⏳ 调拨超时告警**：`inbound_records` 生成超过 **48 小时**未被确认，触发严重告警。

## 6. 异常阻断与系统告警 (Exceptions & Alert Routing)

系统的异常处理遵循“系统告警、人工决策”的解耦原则：

* **工单超时处理**：工单超过 `deadline` 时，系统**只触发告警，不改变单据状态**。调度员介入后，可将异常工单设为 `terminated`。随后调度员需决策：
  * 重新创建一个同阶段的新工单给其他工人。
  * 或在满足条件的情况下，直接触发“手动标记阶段完成”。
* **工人异常完工备注告警路由**：
  * **`normal` (正常)**：静默存盘，无告警。
  * **`damaged` (破损) / `qty_mismatch` (数量不符)**：严重异常。同时推送给**责任调度员**和**管理员**。通知数据需携带 `related_type` 与 `related_id`，支持前端一键跳转至对应工单详情或盘点修正页面。
  * **`other` (其他)**：常规异常。仅推送给**责任调度员**进行关注。

## 7. 技能等级设计 (Skill Level Design)

技能等级为 0-10 之间的整数，0代表不会，数值越大代表技能越熟练。调度算法优先匹配技能等级满足要求且等级差距最小的工人。

### 7.1 调度员手动派单技能口径（新增约束）

对“当前要派发的阶段”计算该订单的阶段技能需求区间：

- `required_skill_min`：该订单所有商品在当前阶段需求技能的最小值
- `required_skill_max`：该订单所有商品在当前阶段需求技能的最大值

派单判定规则：

- **硬拦截**：`worker_skill < required_skill_min` 时，禁止派单（前后端都必须拦截）。
- **风险可放行**：`required_skill_min <= worker_skill < required_skill_max` 时，允许派单但必须走风险确认流程。
- **正常通过**：`worker_skill >= required_skill_max` 时，技能维度无风险。

> 说明：该规则是“阶段维度”的技能判断，不要求工人在其他阶段技能也达标。

### 7.2 工人负载风险口径（新增约束）

调度员派单时，工人负载按 `pending + in_progress` 统计。

- 当 `active_work_order_count >= active_work_order_limit` 时，触发负载风险（`worker_overload`）。
- 负载风险不做硬拦截，允许调度员强制放行。
- `active_work_order_limit` 由配置项控制（默认 5）。

### 7.3 风险放行审计（新增约束）

当派单存在技能风险或负载风险并被强制放行时：

- 调度员必须填写 `override_reason`。
- 系统将风险类型与放行原因写入工单备注结构化前缀：
  - `[override][risk_codes] override_reason`
- 若存在普通备注，普通备注以换行形式追加在结构化前缀后。

工人端工单详情必须能够看到结构化信息（风险类型、放行原因）与备注正文，确保执行环节信息对称。
