# WMS Dispatch — 智能仓储调度管理系统

## 功能概览

| 模块 | 核心能力 |
|------|----------|
| 账号与人员管理 | 多角色账号、工人技能标签、仓库归属（调度员和工人均绑定仓库） |
| 客户管理 | 客户信息维护，订单关联客户 |
| 仓储与商品管理 | 仓库信息（含地理坐标）、商品档案、库存盘点 |
| 订单管理 | 订单创建、调度员接单认领、状态跟踪、明细管理 |
| 工单管理 | 拣货 / 备货 / 发货三类工单，强制串行推进，手动与 AI 双路创建 |
| AI 工单调度 | 自然语言输入，Agent 自动匹配工人与技能，生成工单方案 |
| AI 跨仓调拨 | 自然语言描述库存缺口，Agent 给出调拨建议，走审批流 |
| 效率报告 | 管理员自定义时间范围，LLM 生成图表 + 文字分析报告 |
| 告警通知 | 超时工单、低库存、工单异常备注自动触发站内通知 |

## 角色与权限

系统内置三种角色，权限严格隔离：

- **管理员（Admin）**：管理所有账号、仓库、商品、库存、订单；审批跨仓调拨申请；查看效率报告；终止进行中的工单。
- **调度员（Dispatcher）**：查看本仓库订单并认领接单；创建、管理工单；使用 AI Agent 调度工单与发起调拨；接收告警通知。
- **工人（Worker）**：查看分配给自己的工单；更新工单执行状态；提交完工备注。

## 业务规则

### 订单状态机

订单共四个状态，流转规则如下：

**待接单** → 调度员接单且库存校验通过 → **进行中** → 所有发货工单完成 → **已完成**

**待接单** → 管理员取消 → **已取消**

**进行中** → 管理员逐一终止所有工单后取消 → **已取消**

接单超时（24小时内未创建工单）：系统自动将订单回退为**待接单**，释放预留库存。

### 工单阶段与状态机

**阶段顺序**：拣货 → 备货 → 发货，强制串行。前一阶段全部完成（或由调度员手动标记阶段完成）后，方可创建下一阶段工单。

**同阶段多张工单**：同一阶段可创建多张工单，分配给不同工人并行执行。

**阶段完成的两种路径**：
- 自动完成：该阶段所有工单均流转为已完成状态，阶段自动结束。
- 手动标记：调度员判断已完成工单的成果足够，主动将该阶段标记为完成。前提条件是该阶段至少一张工单已完成，且没有任何进行中的工单（剩余未完成工单须先终止）。阶段一旦标记完成，不可再对该阶段创建新工单。

**工单异常处理**：工单超时只触发告警，不自动改变订单状态。调度员可终止异常工单并重新创建同阶段新工单分配给其他工人，不影响订单继续推进。

**每张工单的状态**：待开始 → 进行中 → 已完成 / 已终止

### 库存状态机

库存分为三个层次：**可用库存**、**预留库存**、**调拨软锁定**。

接单时的库存校验公式：

> 可接单库存 = 可用库存 - 已预留库存 - 调拨软锁定数量

各操作对库存的影响：

- 调度员接单 → 按订单明细全额转入预留库存
- 接单超时回退 / 订单取消 → 释放预留库存
- 所有发货工单完成（订单完成）→ 正式扣减可用库存，释放预留
- 调拨申请提交 → 在来源仓库创建软锁定
- 调拨申请驳回 / 撤回 → 释放软锁定
- 调拨审批通过 → 软锁定转为正式扣减（来源仓库），目标仓库生成待入库记录
- 调度员确认入库 → 目标仓库可用库存正式增加

### 跨仓调拨流程

1. 调度员通过 AI Agent 或手动发起调拨申请，系统对来源仓库该商品创建软锁定。
2. 申请进入管理员审批队列。
3. 管理员审批时，系统二次校验来源仓库库存是否仍满足调拨数量。若库存已不足，提示管理员选择部分调拨或驳回，不允许直接通过。
4. 审批通过后，来源仓库库存立即扣减，软锁定释放，目标仓库生成待入库记录。
5. 目标仓库调度员收到入库通知，实物到达后手动确认入库，库存正式到账。待入库记录超时未确认触发提醒。

## 快速开始

### 环境要求

- Python >= 3.11
- Node.js >= 18
- PostgreSQL >= 15
- 有效的 Qwen API Key

### 安装

```bash
git clone https://github.com/your-org/wms-dispatch.git
cd wms-dispatch
cp .env.example .env
# 填写 .env 中的数据库连接和 API Key

# 启动数据库（需要 Docker）
docker compose up -d

# 后端
cd backend
pip install -r requirements.txt
alembic upgrade head      # 执行数据库迁移
uvicorn main:app --reload # 启动开发服务器

# 前端
cd frontend
npm install
npm run dev
```

### 环境变量

参见 [.env.example](./.env.example)：

```env
DATABASE_URL=postgresql://user:password@localhost:5432/wms
QWEN_API_KEY=your_qwen_api_key
SECRET_KEY=your_secret_key
```

> ⚠️ **安全提示**：`SECRET_KEY` 应使用足够长度的随机字符串（建议 ≥ 32 位），不要使用默认值或弱密钥。

## 安全设计

| 安全领域 | 措施 |
|----------|------|
| 身份认证 | JWT Token（python-jose / bcrypt），Token 有效期控制 |
| 权限控制 | 基于角色的访问控制（RBAC），API 层按角色拦截请求 |
| 密码安全 | bcrypt 哈希存储，禁止明文传输和存储 |
| AI Agent 隔离 | Agent SQL Tool 仅限只读事务，写操作必须走应用 API |
| 输入校验 | Pydantic Schema 校验所有 API 入参，防止注入和非法数据 |
| 通信安全 | 生产环境强制 HTTPS，WebSocket 使用 WSS |

> 详细数据库安全设计见 [database.md](./docs/database.md) 中的「安全设计备注」。
